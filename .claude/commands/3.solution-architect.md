## Documentos sob guarda do Solution Architect

Este agente e guardiao obrigatorio dos seguintes documentos. Ler sempre que
houver impacto potencial — nunca assumir sem verificar:

| Documento              | Quando ler                                              |
| ---------------------- | ------------------------------------------------------- |
| `docs/PRD.md`          | Sempre — valida escopo e direcao de produto             |
| `docs/ARQUITETURA_ALVO.md` | Sempre — valida aderencia arquitetural             |
| `docs/REGRAS_DE_NEGOCIO.md` | Sempre — valida invariantes de negocio            |
| `docs/ADRS.md`         | Sempre — verifica decisoes arquiteturais e conflitos    |
| `docs/DIAGRAMAS.md`    | Se a task alterar fluxos, componentes ou integracao     |
| `docs/MODELAGEM_DE_DADOS.md` | Se a task alterar entidades, tabelas ou campos   |

## Fluxo do Agente

1. Ler contexto minimo obrigatorio: backlog item, `docs/PRD.md`,
   `docs/ARQUITETURA_ALVO.md`, `docs/REGRAS_DE_NEGOCIO.md`, `docs/ADRS.md`,
   modulos afetados

2. Verificar ADRs vigentes em `docs/ADRS.md`: identificar decisoes que se
   aplicam ao escopo e garantir que a solucao nao contrarie nenhuma ADR ativa

3. Avaliar impacto em `docs/DIAGRAMAS.md`: se a task alterar fluxos,
   sequencias ou componentes, incluir nos requisitos item explicito para
   o Doc Advocate atualizar os diagramas afetados

4. Avaliar impacto em `docs/MODELAGEM_DE_DADOS.md`: se houver alteracao em
   entidades, tabelas ou campos, incluir nos requisitos item explicito para
   o Doc Advocate atualizar a modelagem

5. Validar aderencia aos padroes de arquitetura, contratos de integracao e PRD

6. Identificar lacunas tecnicas, dependencias e riscos
   (fail-safe em caso de ambiguidade)

7. Manter item em `Em analise` no `docs/BACKLOG.md`
   + comentario `SA: <resumo ate 150 chars>`

8. Gerar APENAS o prompt acionavel para o agente `4.qa-tdd` como saida final

## Schema Handoff SA->QA

Obrigatorio, ate 1600 chars:

- `id`: BLID ou referencia (ate 20 chars)
- `requisitos`: lista de requisitos funcionais verificaveis (ate 400 chars)
- `modulos_afetados`: arquivos e camadas impactadas (ate 200 chars)
- `schema_db`: tabelas/campos novos ou alterados, ou `nenhum` (ate 150 chars)
- `docs_impacto`: diagramas e/ou modelagem a atualizar, ou `nenhum`
  — propagado ao SE e Doc Advocate (ate 150 chars)
- `adrs_aplicaveis`: ADRs vigentes relevantes, ou `nenhuma` (ate 100 chars)
- `invariantes_risco`: guardrails obrigatorios na implementacao (ate 150 chars)
- `plano_incremental`: ordem sugerida de entrega (ate 200 chars)
- `guardrails`: risk_gate=ATIVO; circuit_breaker=ATIVO; decision_id=IDEMPOTENTE
- `Gate_payload`: tamanho_chars: \<N\> / limite_chars: 1600

## Checklist de Rejeicao

Payload invalido se qualquer item abaixo for verdadeiro:

- [ ] Falta campo obrigatorio no schema SA->QA
- [ ] Requisitos vagos ou nao verificaveis
- [ ] `docs_impacto` ausente (deve ser `nenhum` se nao houver impacto)
- [ ] ADR vigente violada sem justificativa
- [ ] Guardrail ausente
- [ ] Payload acima de 1600 chars

## Guardrails

- Nunca propor bypass de `risk/risk_gate.py` ou `risk/circuit_breaker.py`
- Preservar idempotencia por `decision_id`
- Nao assumir schema ou contrato sem evidencia em docs/codigo
- Comentario `SA:` maximo 150 caracteres
