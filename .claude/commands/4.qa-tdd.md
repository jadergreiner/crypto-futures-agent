Execute o fluxo completo do agente:
1. Analisar requisitos e identificar escopos testaveis
2. Escrever testes unitarios em fase RED (devem FALHAR inicialmente)
   - Nomenclatura: `test_<funcionalidade>_<condicao>_<resultado>`
   - Estrutura AAA: Arrange / Act / Assert
   - Independentes, deterministicos, sem sleeps
3. Cobertura minima: 100% funcoes publicas + integracao critica + risk invariants
4. Atualizar `docs/BACKLOG.md`: status `TESTES_PRONTOS` com referencia da suite
5. Gerar APENAS o prompt executavel para o agente `5.software-engineer` como saida final
   - Incluir: suite completa de testes + contexto + guardrails + plano Green-Refactor + checklist

Schema handoff QA->SE (obrigatorio, ate 2500 chars):
- `id`: BLID ou referencia (ate 20 chars)
- `arquivo_teste`: caminho do arquivo criado (ex: tests/test_<modulo>.py)
- `cobertura`: funcoes publicas cobertas, N testes RED (ate 150 chars)
- `suite_testes`: codigo completo dos testes (bloco unico)
- `requisitos_mapeados`: test_name → requisito (ate 300 chars)
- `invariantes_risco`: guardrails obrigatorios na implementacao (ate 150 chars)
- `plano_green_refactor`: ordem de implementacao sugerida (ate 200 chars)
- `checklist_aceite`: itens verificaveis ao final (ate 150 chars)
- `guardrails`: risk_gate=ATIVO; circuit_breaker=ATIVO; decision_id=IDEMPOTENTE
- `Gate_payload`: tamanho_chars: <N> / limite_chars: 2500

Checklist de rejeicao (payload invalido):
- [ ] Falta campo obrigatorio no schema QA->SE
- [ ] Suite de testes ausente ou incompleta
- [ ] Testes nao estao em fase RED (passam sem implementacao)
- [ ] Guardrail mockado ou ausente
- [ ] Payload acima de 2500 chars

Guardrails:
- Nunca mockar `risk/risk_gate.py` ou `risk/circuit_breaker.py`
- Preservar `decision_id` idempotencia nos testes de decisao/execucao
- Em ambiguidade de comportamento esperado: escolher fail-safe
